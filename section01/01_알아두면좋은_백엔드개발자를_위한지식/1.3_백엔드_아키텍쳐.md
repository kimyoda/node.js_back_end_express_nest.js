# 1.3 백엔드 아키텍처

- 아키텍트: 일반적으로 계획을 작성하는 사람
- 아키텍처: 계획 즉 분석하고 계획하고 정리하는 것
  -> 총체적이고 종합적이고 상세하게 개발을 준비하는 것

- 패턴 중 자주 사용하는 계층형 아키텍처, 이벤트 기반 아키텍처, 마이크로서비스 아키텍처를 알아보고자 한다.

## 계층형 아키텍처

**계층형 아키텍처**(layered architecture)는 레이어드 아키텍처라고 하고 소프트웨어를 몇 가지 계층으로 나누어서 만드는 방식이다.

- 각 계층에서 주어진 역할(프리젠테이션, 비즈니스 로직, 데이터 저장 등)을 수행, 의존성이 단방향이어야 한다.
- 프리젠테이션 계층은 비지니스 계층 / 비지니스 계층은 영속성 계층에만 의존성을 가진다를 의미한다.
- 장점
  - 복잡도가 감소하고, 계층 내에서 관련된 코드의 응집도가 높아진다. 테스트가 쉽다.
- 단점
  - 다른 아키텍처에 비해 애플리케이션 규모가 커지면 계층이 거대해져 유지보수가 힘들고 개발 속도가 떨어진다
  - 큰 덩어리를 여러 개 배포해야 하여 배포도 쉽지 않다.
  - TCP/IP 네트워크 계층은 계층형 아키텍처가 잘 적용된 에다.

예시

```bash
┌──────────────────────────────────────────────────────────────┐
│                    계층형 아키텍처 예시                     │
└──────────────────────────────────────────────────────────────┘

┌────────────────────────────────────┐
│             클라이언트              │
└────────────────────────────────────┘
        ▲ 응답                  │ 요청
        │                      ▼

┌──────────────────────────────────────────────────────────────┐
│  ┌────────────────────────────────────────────────────────┐  │
│  │                 프리젠테이션 레이어                   │  │
│  └────────────────────────────────────────────────────────┘  │
│        ▲ 응답            │ 의존성            │ 요청         │
│        │                ▼                  ▼              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │                   비즈니스 레이어                     │  │
│  └────────────────────────────────────────────────────────┘  │
│        ▲ 응답            │ 의존성            │ 요청         │
│        │                ▼                  ▼              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │             퍼시스턴스(영속성) 레이어                 │  │
│  └────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘

데이터 흐름 : 클라이언트 -> 프리젠테이션 -> 비즈니스 -> 퍼시스턴스
응답 흐름   : 퍼시스턴스 -> 비즈니스 -> 프리젠테이션 -> 클라이언트
의존성      : 프리젠테이션 -> 비즈니스 -> 퍼시스턴스
```

## 이벤트 기반 아키텍처

**이벤트 기반 아키텍처(event-driven architecture)**는 이벤트의 상태 변화에 대응하는 소프트웨어 설계 패턴이다.

- 이벤트는 시스템의 외부 또는 외부에서 시스템에 영향을 주는 상황들을 의미한다.
- 로그인 버튼을 누를 떄 로그인 이벤트 발생 등

- 프로듀서(producer): 이벤트를 발생시키는
- 브로커(broker): 이벤트를 전달하는
- 컨슈머(consumer): 이벤트를 받는

> 수평확장(Scale Out)
> 스케일아웃이라고도 한다. 컴튜팅 장비(서버, 메모리, CPU, 네트워크 등)을 추가해 시스템의 처리량을 늘리는 것을 의미한다.

- 장점
  - 모든 요청을 비동기로 처리한다. 확장성이 좋고 아키텍처 내 컴포너틑 간 의존성을 줄일 수 있다.
  - 프로듀서, 브로커, 컨슈머 각각 수명확장이 용이하다
- 단점
  - 이벤트를 비동기로 처리하여 이벤트 순서를 보장하기 어렵다.
  - 에러 발생가 발생했을 때 이벤트를 새로 받을지, 무시할 지, 에러 처리를 할지 고려해야 한다.

- 예시

```bash
────────────────────────────────────┐
│        이벤트 발생 서비스           │
│   (Producer / 예: 주문 서비스)     │
└────────────────────────────────────┘
                 │
                 │ 이벤트 발행
                 │ (예: OrderCreated)
                 ▼
┌────────────────────────────────────┐
│         이벤트 브로커               │
│   (Kafka / RabbitMQ / Redis)       │
└────────────────────────────────────┘
        │                  │                  │
        │ 이벤트 전달      │ 이벤트 전달      │ 이벤트 전달
        ▼                  ▼                  ▼
┌────────────────┐  ┌────────────────┐  ┌────────────────┐
│   컨슈머 A      │  │   컨슈머 B      │  │   컨슈머 C      │
│   알림 서비스   │  │   재고 서비스   │  │   배송 서비스   │
└────────────────┘  └────────────────┘  └────────────────┘

흐름 : 이벤트 발생 -> 이벤트 브로커 -> 컨슈머 A / 컨슈머 B / 컨슈머 C
```

## 마이크로서비스 아키텍처

**마이크로서비스 아키텍처(microservices architecture)**는 시스템을 여러 개의 작은 서비스로 나누어 관리하는 설계 방식이다.

- 독립적으로 개발, 배포 운영, 구조적으로 분리되어있다.
- 각 서비스는 독립적으로 업데이트 및 스케일링이 가능, 시스템 전체의 구조는 간결해진다.
- 주의할 점은 하나의 서비스를 너무 잘게 쪼개는 것이다.

- 마이크로서비스 아키텍처 자체는 도메인 주도 설계(domain-driven design)에 영향을 많이 받는다.
- 도메인은 요구하는 문제 분야 내에서 상황이나 내용을 말한다. 예를 들어 계정관리, 쇼핑몰 상품 관리 등이다.
- 하나의 서비스가 하나의 도메인을 담당하는 것이 좋다.

- 장점
  - 확장성이 좋고, 업그레이드가 용이하다.
  - 서비스별로 확장하거나 서비스만 따로 업그레이드할 수 있기 때문이다.
- 단점
  - 트랜잭션을 처리하려면 분산 트랜잭션 등을 고려해야 하여 난이도가 올라간다.
  - 하나의 서비스를 운영하는 모든 구성을 갖춰야 하여 유지보수 하는데 비용이 많이 들어간다.

- 예시

```bash
┌──────────────────────────────────────────────────────────────┐
│                마이크로서비스 아키텍처 예시                  │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                        클라이언트                            │
│                                                              │
│   ┌──────────┐      ┌──────────┐      ┌──────────┐          │
│   │   웹     │      │  모바일  │      │   PC     │          │
│   └──────────┘      └──────────┘      └──────────┘          │
└──────────────────────────────────────────────────────────────┘
                              │
                              │ 요청 / 응답
                              ▼
┌──────────────────────────────────────────────────────────────┐
│                           백엔드                             │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │                    API 게이트웨이                     │  │
│  │                                                        │  │
│  │  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐      │  │
│  │  │  인증  │  │ 라우팅 │  │ 로깅   │  │ 집계   │      │  │
│  │  └────────┘  └────────┘  └────────┘  └────────┘      │  │
│  └────────────────────────────────────────────────────────┘  │
│       ↕                ↕                ↕                ↕   │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐  ┌────────┐ │
│  │ 사용자     │  │ 주문       │  │ 결제       │  │ 알림   │ │
│  │ 서비스     │  │ 서비스     │  │ 서비스     │  │ 서비스 │ │
│  └────────────┘  └────────────┘  └────────────┘  └────────┘ │
└──────────────────────────────────────────────────────────────┘

화살표는 요청과 응답을 의미합니다.
```

-> 이 외에도 파이프라인 아키텍처(pipeline architecture), 마이크로커널 아키텍처(microkernel architecture), 서비스 지향 아키텍처(service oriented architecture), 공간 기반 아키텍처(space-based architecture) 등이 있다.

## 파이프라인 아키텍처

**파이프라인 아키텍처(pipeline architecture)**는 데이터를 여러 처리 단계를 순서대로 통과시키며 변환하는 설계 방식이다.

- 각 처리단계를 **필터(filter)**라고 하고, 단계간 데이터를 전달하는 통로를 **파이프(pipe)**라고 한다.
- 각 필터는 독립적으로 동작, 앞 단계의 출력이 다음 단계의 입력이 된다.
- Unix의 `I` 명령어, ETL 데이터 파이프라인, CI/CD 파이프라인 등이 예시다.
- 장점
  - 각 필터가 독립적이어서 재사용성과 교체가 용이하다.
  - 처리 흐름이 단방향으로 명확하여 이해하기 쉽다.
  - 필터를 병렬로 실행하면 성능을 향상시킬 수 있다.
- 단점
  - 필터 간 데이터 형식을 맞춰야하여 인터페이스 설계가 중요하다
  - 중간 단계에서 오류 발생 시 전체 파이프라인이 영향을 받을 수 있다
  - 단계가 많아질수록 지연이 누적될 수 있다.

- 예시

```bash
┌──────────────────────────────────────────────────────────────────────┐
│                    파이프라인 아키텍처 예시 (ETL)                    │
└──────────────────────────────────────────────────────────────────────┘

┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  입력     │     │ 필터 1   │     │ 필터 2   │     │ 필터 3   │     │  출력     │
│ (원본     │────▶│ (추출)   │────▶│ (변환)   │────▶│ (적재)   │────▶│ (결과    │
│  데이터)  │pipe │ Extract  │pipe │Transform │pipe │  Load    │pipe │  저장)   │
└──────────┘     └──────────┘     └──────────┘     └──────────┘     └──────────┘

흐름 : 입력 데이터 -> 추출(Extract) -> 변환(Transform) -> 적재(Load) -> 출력
특징 : 각 필터는 독립적으로 동작하며, 파이프를 통해 다음 단계로 데이터를 전달
```

## 마이크로커널 아키텍처

**마이크로커널 아키텍처(microkernel architecture)** 핵심 기능만을 담은 **코어 시스템(core system)**과 확장 기능을 담당하는 **플러그인(plugin)**으로 분리하는 설계방식이다.

- 코어 시스템은 최소한의 기능만 제공, 나머지 기능은 플러그인으로 추가한다.
- 플러그인은 독립적으로 개발, 배포, 제거가 가능하다
- IDE, VS Code 확장 시스템, 운영체제 커널 구조

- 장점
  - 새로운 기능을 플러그인으로 쉽게 추가할 수 있다
  - 코어 시스템이 단순하여 안전성이 높고 테스트가 용이하다
  - 플러그인을 독립적으로 업데이트하거나 교체할 수 있다
- 단점
  - 플러그인 간 의존성이 생기면 관리가 복잡해진다
  - 코어 시스템의 설계가 잘못되면 전체 시스템에 영향을 미친다
  - 플러그인 간 통신 비용이 발생할 수 있다
- 예시

```bash
┌──────────────────────────────────────────────────────────────┐
│                   마이크로커널 아키텍처 예시                  │
└──────────────────────────────────────────────────────────────┘

              ┌───────────────────────────────┐
              │         코어 시스템            │
              │  (최소한의 핵심 기능만 포함)   │
              │  - 기본 라우팅                 │
              │  - 플러그인 관리               │
              │  - 공통 인터페이스             │
              └───────────────────────────────┘
                  │         │         │
        ┌─────────┘         │         └─────────┐
        ▼                   ▼                   ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│   플러그인 A   │  │   플러그인 B   │  │   플러그인 C   │
│  (결제 기능)   │  │  (알림 기능)   │  │  (분석 기능)   │
└───────────────┘  └───────────────┘  └───────────────┘

특징 : 플러그인은 독립적으로 추가/제거 가능, 코어 시스템은 변경 최소화
```

## 서비스지향 아키텍처

**서비스 지향 아키텍처(service oriented architecture, SOA)** 시스템의 기능을 독립적인 서비스 단위로 나눈다
이를 네트워크를 통해 연동하는 설계 방식

- 마이크로서비스 아키텍처 전신, 기업 내 여러 시스템을 서비스 단위로 통합하는데 주로 사용된다
- 서비스간 통신은 **ESB(Enterprise Serivce Bus)**를 통해 이루어지는 경우가 많다.
- ESB는 서비스 간 메시지 변환, 라우팅, 프로토콜 변환 등을 담당하는 미들웨어다.
- SOA는 서비스 크기가 상대적으로 크고 ESB에 의존하나, 마이크로서비스는 더 작고 독립적이다.

- 장점
  - 기업 내 기존 레거시 시스템을 서비스로 감싸 재사용할 수 있다
  - 서비스 단위로 개발, 배포가 가능하여 유연성이 높다
  - 표준 프로토콜을 사용해 시스템 간 통합이 용이하다.

- 단점
  - ESB가 병목 지점이 될 수 있고, ESB 자체가 단일 장애점이 될 수 있다
  - 서비스 간 통신이 복잡해질수록 관리 비용이 증가한다
  - 서비스 단위가 크고 무거워 민첩성이 떨어질 수 있다.

- 예시

```bash
┌──────────────────────────────────────────────────────────────┐
│                  서비스 지향 아키텍처 예시                    │
└──────────────────────────────────────────────────────────────┘

┌───────────┐   ┌───────────┐   ┌───────────┐   ┌───────────┐
│클라이언트 A│   │클라이언트 B│   │클라이언트 C│   │클라이언트 D│
└───────────┘   └───────────┘   └───────────┘   └───────────┘
       │               │               │               │
       └───────────────┴───────────────┴───────────────┘
                               │
                               ▼
       ┌────────────────────────────────────────────┐
       │         ESB (Enterprise Service Bus)       │
       │  - 메시지 라우팅 / 프로토콜 변환           │
       │  - 서비스 오케스트레이션                   │
       └────────────────────────────────────────────┘
              │              │              │
              ▼              ▼              ▼
       ┌────────────┐ ┌────────────┐ ┌────────────┐
       │ 서비스 A    │ │ 서비스 B    │ │ 서비스 C    │
       │ (회원관리)  │ │ (결제)     │ │ (재고관리)  │
       └────────────┘ └────────────┘ └────────────┘

특징 : ESB를 중심으로 서비스들이 연결되며, 메시지 변환과 라우팅을 ESB가 담당
```

## 공간 기반 아키텍처

**공간 기반 아키텍처(space-based architecture)** 데이터와 처리 로직을 **인메모리 공유 공간**에 분산배치하여 높은 확장성과 성능을 달성하는 설계 방식이다

- 데이터베이스나 중앙 서버에 대한 병목을 없애기 위해, 데이터를 메모리에 분산 저장, 처리 유닛들이 공유 공간을 통해 데이터를 주고받는다.
- 대규모 트래픽이나 동시 사용자가 급격히 증가하는 시스템(티켓 예매, 경매 시스템)에 적합하다
- 주요 구성 요쇠 처리 유닛, 가상화된 미들웨어, 데이터 펌프, 데이터 라이터, 데이터 리더

- 장점
  - 공유 메모리 기반으로 데이터베이스 병목을 제거, 매우 높은 확장성을 가진다
  - 처리 유닛을 동적으로 추가하여 트래픽 급중에 유연하게 대응할 수 있다
  - 응답 속도가 빠르고 고가용성을 확보하기 용이하다
- 단점
  - 인메모리 데이터 특성상 장애 발생 시 데이터 유실 위허미 있다.
  - 아키텍처 설계와 구현이 복잡하고 운영 비용이 높다
  - 데이터 일관성 보장이 어렵다

- 예시

```bash
┌──────────────────────────────────────────────────────────────┐
│                    공간 기반 아키텍처 예시                    │
└──────────────────────────────────────────────────────────────┘

┌───────────┐   ┌───────────┐   ┌───────────┐
│  사용자 A  │   │  사용자 B  │   │  사용자 C  │
└───────────┘   └───────────┘   └───────────┘
       │               │               │
       └───────────────┴───────────────┘
                       │
                       ▼
       ┌───────────────────────────────────────┐
       │         가상화된 미들웨어               │
       │  (메시징 그리드 / 데이터 그리드 등)    │
       └───────────────────────────────────────┘
          │              │              │
          ▼              ▼              ▼
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │  처리 유닛 1  │ │  처리 유닛 2  │ │  처리 유닛 3  │
  │ (인메모리 DB) │ │ (인메모리 DB) │ │ (인메모리 DB) │
  └──────────────┘ └──────────────┘ └──────────────┘
          │              │              │
          └──────────────┴──────────────┘
                         │ 데이터 동기화
                         ▼
             ┌───────────────────────┐
             │       영속성 DB        │
             │  (비동기 데이터 저장)  │
             └───────────────────────┘

특징 : 처리 유닛이 각각 인메모리 DB를 보유, 트래픽 증가 시 처리 유닛을 동적으로 추가
```
