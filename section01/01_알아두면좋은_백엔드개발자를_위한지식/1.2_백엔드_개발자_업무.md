# 1.2 백엔드 개발자 업무

1. 과제 할당
2. 과제 분석
3. 개발
4. 테스트
5. QA
6. 배포
7. 유지보수 순서로 진행된다.

-> 개발은 `분석  -> 개발 -> 테스트` 사이클을 반복하면서 완성도를 높여 진행된다.
동료나 사수분의 리뷰를 받으면서 코드의 완성도를 높여 코드에 추가한다.

- 개발하면서 개발자 스스로 테스트를 해봐야 한다.
- 이러한 방식을 `테스트 주도 개발` 이라한다.

  > 테스트 주도 개발(Test-Driven-Developmenet, TDD)
  > 개발자가 기능을 구현하기 전에 테스트를 먼저 만들고, 코드를 구현하는 개발 방법론이다. 테스트 작성과 코드 구현의 사이클을 반복해가며 개선 하는것이 목표이다.

- 끝으로는 배포 방법이 있다.
- 롤링 배포: 특정 개수의 서버들을 돌아가며 배포
- 블루 그른 배포: 똑같은 서버 인스턴스를 띄우고 로드밸런서 설정을 바꿔버리는 것
- 카나리 배포: 전체 서버의 특정 비율만큼 배포하고 문제 없는 경우 배포를 늘려나가는 경우

- 이외에 도커 같은 컨테이너(소프트으웨어의 실행에 필요한 실행 환경을 독립적으로 운용할 수 있도록 기반 환경 또는 다른 실행 환경과의 간섭을 막고 실행의 독립성을 확보해주는 운영체제 수준의 격릭 기술) 배포가 훨씬 쉬워졌다.
- 쿠버네티스(k8s)를 사용하여 배포와 관리를 쉽게 할 수 있는 장점이 있다. -> 서버를 돌아가며 배포하는 롤링 배포도 앤서블을 사용하면 손쉽게 구현할 수 있다.

- 위의 배포 과정을 CI(Continuous Integration)/CD(Continuous Delivery & Continuous Deployment)라고 한다.
- CI는 지속적인 통합 -> 새로운 코드 변경 사항이 정기적으로 빌드 및 테스트되어 저장소에 통합 -> 저장소에 코드가 반영
- CD는 지소적인 배포를 말한다. 즉, 실제 프로덕션 환경까지 변경된 코드가 반영

- 위의 업무를 전담하는 Devops(데브옵스)업무도 있다.
- 개발과 운영의 합성이고, 주로 클라우드 마이그레이션, 데이터 파이프라인 작성, 모니터링 시스템 구축, 코드를 통한 인프라 관리 등을 담당한다.
